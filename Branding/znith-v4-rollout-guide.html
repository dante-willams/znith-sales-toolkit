Rule · Branch strategy Create feat/znith-v4 from current branch. All work lands there. User will review the preview deploy before merging to main. Never force-push.

Order matters. Do steps 1–4 across all files first (they're mechanical and low-risk), then 5–8 sequentially per file (they require visual review), then 9 last.

02Step 1 · Extract tokens to a shared file

Currently each app redefines the :root token block. This has caused drift (v1 files use #0e0d0b, v3 files use #05070F). Consolidate.

  1. Create shared/znith-tokens.css with the v4 token block from the branding spec (both dark + light).
  2. In each app's HTML, add <link rel="stylesheet" href="../shared/znith-tokens.css"> at the top of <head> (use ./shared/ for dashboard.html).
  3. Remove the duplicate :root blocks from inline <style>.
  4. Keep any app-specific tokens (e.g. a chart palette unique to value-map) in the local :root — just remove the system-wide ones.

06Step 5 · Contrast fixes

The v3 --tfaint at 0.28 opacity fails WCAG AA. Bump everywhere to 0.42 (already in shared tokens from step 1). Then audit usages:

  1. Grep each app for tfaint. If it is used on text smaller than 12px, either bump to --tmuted or enlarge to 13px.
  2. Replace decorative-only uses (axis labels, dot patterns, chrome backgrounds) with --tghost.
  3. Check all rgba(255,255,255,0.XX) inline values. Anything below 0.42 for text → replace with --tghost or bump up.

07Step 6 · Brighten amber

Global find-and-replace:

#B8860B  →  #E5A83A
#d4a012  →  #B8860B   (hover/pressed — now uses old amber)
rgba(184,134,11,0.22)  →  rgba(229,168,58,0.22)
rgba(184,134,11,0.12)  →  rgba(229,168,58,0.12)
rgba(184,134,11,0.07)  →  rgba(229,168,58,0.06)
Rule · Amber buttonsOn amber-fill buttons (e.g. .btn-new-deal), the text color should be #0A0E1A (surface-2) — keep that. Do not change button text to white; amber is too light for white text at AA.

08Step 7 · Number treatment

Apply the number-treatment primitives to any KPI, metric, score, or tabular number:

  1. Add font-variant-numeric: tabular-nums; to body in the shared token file (already done).
  2. Audit hero numbers. If rendered as "$24.8M" in one span, split into three spans: 
    <span class="kpi-value">
  <span class="prefix">$</span>
  <span class="num">24.8</span>
  <span class="unit">M</span>
</span>
  3. Add the delta-chip component anywhere ± values appear (MEDDPICC changes, forecast deltas, etc.). Reference implementation: znith-v31-prototype.html.

09Step 8 · Light theme rebuild

The current [data-theme="light"] block in dashboard.html is a minimal token swap. It needs to become a full parallel design.

Per-app tasks:

  1. Remove the local [data-theme="light"] block (it is now in shared/znith-tokens.css).
  2. Audit every inline box-shadow or filter: drop-shadow rule. Dark glows don't translate to light. Wrap in theme-aware form: 
    /* Before */
box-shadow: 0 0 8px var(--amber-glow);

/* After */
box-shadow: var(--sh-amber);

/* And in shared tokens: */
--sh-amber: 0 0 8px var(--amber-glow);         /* dark: glow */
[data-theme="light"] { --sh-amber: 0 2px 12px rgba(201,140,32,0.24); } /* light: shadow */
  3. Audit backdrop-filter: blur() usages. They work on both themes but the rgba background must be theme-paired.
  4. Test the amber accents. On light, --amber (#C98C20) is used for fills; --amber-text (#946C00) for text. Make sure buttons use the right variable.

12Verification protocol

For each app:

1

Local screenshot — dark

Open the app. Screenshot. Compare to znith-v31-prototype.html for chrome consistency (header height, footer border, amber accent color).

2

Local screenshot — light

Toggle theme. Verify no glow-shadow regression, amber reads dark-gold not bright-yellow, borders are solid not translucent.

3

Contrast audit

Use Chrome DevTools → Lighthouse → Accessibility. Aim for zero AA contrast failures. Document any accepted failures (charts, brand-required) with a comment.

4

Click-through

From dashboard.html, create a test deal. Navigate into each sub-tool. Use the back button. Ensure the ?deal= query parameter is preserved and tools still load data correctly.

5

Deploy to preview

Push feat/znith-v4 and trigger the preview deploy. Share URL with user. Do not merge to main.

13Known pitfalls

Clip-path + focus rings outline and box-shadow-based focus rings get clipped by clip-path. For any input, select, textarea — do not chamfer. Use border-radius: 3px and a proper focus ring instead.
Clip-path + dropdowns A chamfered card cannot render a dropdown that escapes its bounds — clip-path forces overflow: hidden effectively. If a tool has a dropdown inside a chamfered card, either (a) portal the dropdown to body, or (b) remove the chamfer from that specific card.
Sticky header + backdrop-filter Some Chromium versions composite sticky + backdrop-filter incorrectly on scroll. If the header flickers, add transform: translateZ(0) to force its own layer.
localStorage theme key Current code uses localStorage.getItem('znith-theme'). Keep this key — do not rename it to conga-theme. Users have existing values stored and renaming breaks their preference.
Safe to touch All styling, all HTML chrome, all inline SVG paths in headers.
Do not touch shared/dealStore.js data model. auth.js. Any ?deal= query parameter handling. Netlify/Vercel config files.
