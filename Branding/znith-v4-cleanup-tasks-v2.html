00Why this drift exists (context for Claude Code)

Each tool was built as a standalone page with its own <style> block and color vocabulary before the v4 spec existed. The brand-refresh pass added the shared tokens file, the Conga mark, the title rename, and the footer without going back to sweep inline style blocks. So every tool now uses v4 chrome (tokens link, header, footer) but carries v3-era guts (inline hex stage colors, var(--green) references that point nowhere, dark-only glow shadows). This cleanup closes that gap.

The other reason: the shared tokens file renamed the status colors (--green → --status-ok) but the v1-compatibility aliases were not added, so every pre-existing var(--green) reference silently broke. Task CLN-01 fixes this with a single edit.

02CLN-01 · Add v1 color aliases to shared tokens

CLN-01 Unblock 93 broken var(--green)-style references P0
Files: shared/znith-tokens.css
Problem: 4 tools (win-room, account-brief-v2, value-map, prospecting-agent) collectively reference var(--green), var(--yellow), var(--red), var(--blue), and their -muted variants 93 times. The shared tokens file defines --status-ok etc. but not these aliases, so every reference resolves to unset and renders as default/black.
Fix: Append aliases to the dark :root block AND the [data-theme="light"] block so both themes resolve. This should already be present — verify and add if missing.
/* shared/znith-tokens.css — inside :root (dark, default) */
--green:        var(--status-ok);
--yellow:       var(--status-warn);
--red:          var(--status-risk);
--blue:         #63B3ED;
--green-muted:  rgba(52,211,153,0.14);
--yellow-muted: rgba(245,158,11,0.14);
--red-muted:    rgba(239,68,68,0.14);
--blue-muted:   rgba(99,179,237,0.14);

/* shared/znith-tokens.css — inside [data-theme="light"] block */
--green:        #16A34A;
--yellow:       #B47814;
--red:          #C83232;
--blue:         #2B6FAE;
--green-muted:  rgba(22,163,74,0.12);
--yellow-muted: rgba(180,120,20,0.12);
--red-muted:    rgba(200,50,50,0.10);
--blue-muted:   rgba(43,111,174,0.10);

Verify: Open each of the 4 affected tools in browser devtools, inspect any health pill or status indicator, confirm computed color is populated (not default black text).

04CLN-03 · Add stage color tokens to shared

CLN-03 Eliminate 44 hardcoded stage hex values across 6 tools P1
Files: shared/znith-tokens.css + find/replace in dashboard, win-room, account-brief-v2, value-map, qualify-iq, demo-brief
Problem: Stage pill colors are hardcoded inline in every deal-showing tool: #63B3ED (discovery), #A78BFA (proposal), #FB923C (negotiation), #34D399 (closed-won). If the palette ever shifts, we're touching 6 files.

--stage-proposal:    #A78BFA;
--stage-negotiation: #FB923C;
--stage-closed-won:  var(--status-ok);

Step 2 — find/replace across the 6 files listed above:

#63B3ED  →  var(--stage-discovery)
#A78BFA  →  var(--stage-proposal)
#FB923C  →  var(--stage-negotiation)
#34D399  →  var(--stage-closed-won)

Exception: leave the hex values inside rgba(99,179,237,…), rgba(167,139,250,…), etc. — those are muted backgrounds. You'll handle those in CLN-05 if they matter. For this task, only replace bare hex codes in color:, background:, border-color:, fill:, and stroke: declarations.

06CLN-05 · Light-theme rebuild for value-map

CLN-05 value-map has zero light-theme rules — toggle produces broken UI P0
Files: value-map/index.html
Problem: grep "[data-theme=\"light\"]" returns 0 hits in value-map. The shared tokens handle the palette but any local rgba() overrides, glows, and dark-only backgrounds will persist when the user toggles light. Given value-map is 304 KB of tool logic with charts and custom panels, local overrides definitely exist.
Fix: Pattern-match what win-room and dashboard do. At minimum:
  • Audit every inline rgba(255,255,255,…) — these are dark-mode only. Replace with var(--border), var(--surf-2), etc. where semantic, or wrap in a [data-theme="light"] override.
  • Audit every background: #0… or color: rgba(255,255,255,…) — replace with tokens.
  • Add a [data-theme="light"] block at the end of the <style> for topbar rgba(255,255,255,0.94) override and any chart color swaps.
  • Test every screen of value-map in both themes. Chart fills and axis labels are usually the worst offenders.

Reference: copy the structure from win-room/index.html lines at the bottom of the <style> block — the 3 [data-theme="light"] rules there are the minimum baseline.

07CLN-06 · Light-theme audit for 5 "thin" tools

CLN-06 Most tools only have 1 light-mode override — extend coverage P1
Files: account-brief-v2, demo-brief, prospecting-agent, qualify-iq, sales-hub
Problem: These 5 tools have only a single [data-theme="light"] rule (typically .topbar { background: rgba(255,255,255,0.94); }). That's not enough — any tool-specific card, modal, toast, drawer, or chart element that uses a local rgba/hex will break on light.
Fix — apply to each tool:
  • Toggle theme in browser, walk through every screen/state in the tool.
  • For each visual issue, find the offending CSS rule. Prefer fixing the root cause (replace hex with token). If the rule must be theme-specific, add a [data-theme="light"] override at the bottom of the <style> block.
  • Common offenders: modal backdrops, dropdown menus, code/mono blocks, data-table zebras, tooltip backgrounds, focus rings.
  • Re-check every box-shadow — dark-mode glows almost always need replacement with var(--sh-sm), var(--sh-md), var(--sh-lg), or status variants.

Acceptance: cycle theme on each tool, every screen renders cleanly with no white-on-white, no black-on-black, no colored halos, no chart axis labels disappearing.

08CLN-07 · Root-level font-variant-numeric scoping

CLN-07 Tabular nums are forced globally — should be per-numeric-element P2
Files: shared/znith-tokens.css
Problem: The shared tokens file sets font-variant-numeric: tabular-nums on numeric display classes, but also on body in several tool files. When applied to prose, numbers inside paragraphs get unnaturally wide spacing. Confirm the shared rule is scoped correctly.
Fix: Verify shared/znith-tokens.css only applies tabular-nums to:
.kpi-value, .kpi-value *,
.meddpicc-score-val, .drawer-value-chip-val, .drawer-dim-score,
.meta-val, .score, [data-num] {
  font-variant-numeric: tabular-nums;
}

Then grep each tool for body { … font-variant-numeric: tabular-nums … } and remove. Prose body copy should use proportional figures by default.

CLN-08 Standardize prefix / num / unit pattern across tools P2
Files: verify all tools use shared classes from znith-tokens.css
Problem: KPI numbers across tools show inconsistent scale relationships between prefix ($), the number, and the unit (M, K, %). The shared tokens already define the right component — confirm every tool uses it.
