01Critical — broken CSS variables

CLN-01 Fix unresolved v1 color variables in win-room P0 · Blocking
Files: win-room/index.html (and likely all other apps — grep first)
Problem: Win Room references var(--green), var(--yellow), var(--red), var(--blue), var(--green-muted), var(--yellow-muted), var(--red-muted). These were v1 tokens. The v4 shared tokens define --status-ok, --status-warn, --status-risk. The v1 names do not resolve — health pills, gap severity bars, MEDDPICC indicators, stakeholder roles, "done" states are all rendering with their fallback (often transparent or inherited colors).

Fix — choose ONE approach and apply consistently:

Option A (preferred) · add aliases to the shared tokens file so existing code keeps working:

/* Append to shared/znith-tokens.css in the :root block */
--green:        var(--status-ok);
--yellow:       var(--status-warn);
--red:          var(--status-risk);
--blue:         #63B3ED;
--green-muted:  rgba(52,211,153,0.14);
--yellow-muted: rgba(245,158,11,0.14);
--red-muted:    rgba(239,68,68,0.14);

/* And in [data-theme="light"] block, re-point muted variants: */
--green-muted:  rgba(22,163,74,0.12);
--yellow-muted: rgba(180,120,20,0.12);
--red-muted:    rgba(200,50,50,0.10);

Option B · find-and-replace across all apps. Slower but cleaner. If you take this route:

var(--green)        →  var(--status-ok)
var(--yellow)       →  var(--status-warn)
var(--red)          →  var(--status-risk)
var(--blue)         →  #63B3ED         /* no v4 token — define one if reused */
var(--green-muted)  →  rgba(52,211,153,0.14)
var(--yellow-muted) →  rgba(245,158,11,0.14)
var(--red-muted)    →  rgba(239,68,68,0.14)

Pick Option A unless you have time for a full sweep. Either way, run grep -rn "var(--green)\|var(--yellow)\|var(--red)\b\|var(--blue)" --include="*.html" across the repo to find all occurrences and verify nothing is missed.

Files: win-room/index.html, likely others

--sh-risk: 0 1px 3px rgba(200,50,50,0.2);

/* Then in win-room and others: */
.ind-green { box-shadow: 0 0 6px rgba(22,163,74,0.4); }

04Amber hygiene

CLN-05 Stop spraying amber on passive text in win-room P1
Files: win-room/index.html
Problem: .stakeholder-role renders static labels like "DECISION MAKER" in var(--amber). Amber is the system voice / active-state color. Using it for passive metadata dilutes the signal and breaks the branding spec's "amber = charged/acting only" rule.
.stakeholder-role { ...; color: var(--amber); }
.stakeholder-role { ...; color: var(--tmuted); }

Audit the rest of win-room for similar misuse: any static label, eyebrow, or decorative text in var(--amber) that isn't representing a live / active / CTA state should drop to var(--tmuted) or var(--tfaint).

Amber belongs on: section eyebrows (scannable structural labels), CTAs, active tabs, the footer mark, live pips, charged numbers. Nowhere else.

// Before

// After — swap

Remove the JS line that sets .textContent of the toggle. The SVGs handle their own visibility.

CLN-07 Remove emoji from empty states P2
Files: dashboard.html, win-room/index.html, likely others
Problem: Empty states use ◈ (a Unicode diamond) and ⚠. The diamond is acceptable as a single geometric glyph, but ⚠ renders as a colored emoji on macOS/iOS and breaks the monochrome tone.
<div class="state-icon">⚠</div>
<div class="state-icon">
  <svg width="40" height="40" viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="1.5">
    <path d="M10.29 3.86L1.82 18a2 2 0 0 0 1.71 3h16.94a2 2 0 0 0 1.71-3L13.71 3.86a2 2 0 0 0-3.42 0z"/><line x1="12" y1="9" x2="12" y2="13"/><line x1="12" y1="17" x2="12.01" y2="17"/>
  </svg>
</div>

Keep ◈ — it's geometrically on-brand and renders as text not emoji. Or replace it with a small oscilloscope-trace SVG for the true Znith empty state from the v4 spec.

CLN-09 Apply tabular-nums to number display, not the whole body P2
Files: shared/znith-tokens.css
Problem: font-variant-numeric: tabular-nums is declared on :root, which forces tabular figures on EVERYTHING — including prose paragraphs and UI copy. This makes all non-numeric text feel slightly mechanical. The intent (from the v4 spec) is tabular for numbers, proportional for text.
/* Remove from :root */
font-variant-numeric: tabular-nums;

/* Instead, target number-bearing classes */
.kpi-value, .kpi-value *, .meddpicc-score-val, .drawer-value-chip-val,
.drawer-dim-score, .meta-val, .score, [data-num] {
  font-variant-numeric: tabular-nums;
}

Grep for hero numbers, score displays, meta rows showing dollar values, timers — add the selector to the list above.

06Nice-to-haves that close out the spec

CLN-10 Adopt number-treatment markup on hero KPIs P2
Files: any tool with hero metrics (dashboard.html deal cards show $, win-room health pills, sales-hub KPIs, value-map ROI chips)
Problem: Numbers render as single spans — $24.8M all in one piece. The v4 spec prescribes prefix + number + unit at three typographic weights/sizes for editorial credibility.
/* Markup pattern */
<span class="kpi-value">
  <span class="prefix">$</span>
  <span class="num">24.8</span>
  <span class="unit">M</span>
</span>

/* Shared CSS — add once to znith-tokens.css or a new shared/znith-components.css */
.kpi-value { font-family: var(--f-brand); font-weight: 800; letter-spacing: -0.04em; font-variant-numeric: tabular-nums; }
.kpi-value .prefix,
.kpi-value .unit { font-size: 0.5em; opacity: 0.55; font-weight: 600; margin: 0 0.08em; }
.kpi-value .num    { font-size: 1em; }

Apply to the most visually prominent numbers only — deal-value on deal cards, KPI hero values, ROI totals in value-map. Small meta numbers (row ages, counts) stay as-is.
