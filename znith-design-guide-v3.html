04
Geometry — The Chamfer System
Every UI element uses chamfered corners via CSS clip-path — not border-radius. This is the defining geometric signature of the toolkit. Apply it consistently.
--clip-lg
10px cut
cards · modals · panels
--clip-sm
5px cut
buttons · badges · inputs
border-radius
❌ Do not use
✓ Apply clip-path to
→ All card / panel containers (clip-lg)
→ All buttons, primary and ghost (clip-sm)
→ All form inputs and textareas (clip-sm)
→ All badges and status pills (clip-sm)
→ Modals and drawer panels (clip-lg)
→ Step number indicators (clip-sm)
→ Code blocks and pre elements (clip-lg)
✗ Do not use chamfer on
→ The header or footer (these span full width)
→ Dividers and horizontal rules
→ Circular elements (pips, avatar dots)
→ SVG icons
→ body or html elements
→ Any element with overflow:visible children that need to escape
Never
Do not use border-radius as a substitute for clip-path on interactive elements. These are not interchangeable — border-radius creates a fundamentally different visual language. The exception: pure circles (4px pip dots use border-radius:50%).

06
UI Switcher — Step Progress Dots
Multi-step tool pages (Account Brief, Qualify IQ, Demo Brief, etc.) use step progress dots in the header-right area. This is the canonical navigation metaphor — do not replace it with numbered circles, tab bars, or text-based step indicators.
Step Dots — All States
Step 1 active, 2–3 pending
Step 1 done, step 2 active
Steps 1–2 done, step 3 active
step-dot CSS
.step-dot { width:8px; height:8px; border-radius:50%; background:var(--border-md); } .step-dot.active { background:var(--amber); box-shadow:0 0 6px var(--amber-glow); } .step-dot.done { background:var(--amber); opacity:0.55; }
Never
Do not replace step dots with numbered circles (e.g. "① ② ③"), text tabs, or pill nav bars. Step dots are the correct, compact metaphor for in-header progress — they preserve header space while communicating position. If a tool has more than 5 steps, discuss before adding more dots.

Spacing, max-width, and padding values that create consistent rhythm across all pages. Use these as constraints — not suggestions.

Card gap (grids)

Section spacing (between content blocks)
margin-top: 40–52px Between distinct content sections
Card internal padding
padding: 20–24px Standard card body
Rule
Always set body padding-bottom ≥ 100px to prevent content from hiding behind the fixed /Znith footer. Always set the main content container to max-width 860–960px centered — never full-width on wide screens. The 8px grid underlies all spacing — use multiples of 8 for gap, margin, and padding values.
08
Do / Don't — Design Decisions
Common mistakes observed across the current toolkit pages. These have already degraded visual consistency — do not repeat them in new work.
✓ Do
→ Use amber for all primary interactive states — buttons, focus rings, metric values, step dots
→ Use clip-path chamfers on every interactive and card element
→ Use IBM Plex Mono for all labels, eyebrows, and metadata — never Barlow for these
→ Keep the header exactly 64px tall, sticky, with glassmorphism background
→ Let dark surfaces dominate — content sits on --surf (#0F1520), not raw --bg
→ Use step dots in header-right for multi-step flows — this is the only navigation metaphor
→ Match transition timing to --transition (220ms ease) on all hover states
→ Use --c-[module]-muted as badge backgrounds — never full saturation fills
→ Keep section eyebrows in mono · 9px · 0.18em LS · amber
✗ Don't
→ Don't use module accent colors (--c-account, etc.) as dominant fill colors
→ Don't alias color tokens to wrong names (e.g. --gold: #8B5CF6 — this is purple, not gold)
→ Don't vary header height between pages — it must be exactly 64px everywhere
→ Don't use border-radius on buttons, cards, or inputs — use clip-path only
→ Don't add DM Mono or any other monospace font — IBM Plex Mono is locked
→ Don't create page-specific accent colors not in the token system
→ Don't use different step/nav metaphors per tool (numbered circles, tab bars, pill navs)
→ Don't set --transition: all 0.15s ease — use the canonical 220ms cubic-bezier
→ Don't skip padding-bottom on body — content will hide behind the footer

Design Debt — Known Issues to Fix
These inconsistencies exist today in the toolkit (as of claude/funny-haibt-6098d6). When editing any of these pages, fix the issue in the same pass — do not leave debt in place.
High
qualify-iq and demo-brief use DM Mono instead of IBM Plex Mono
These pages import DM Mono and use it for footer/badge mono text. Replace with IBM Plex Mono and remove the DM Mono import. Also correct border-top: 1px solid rgba(249,115,22,0.18) in demo-brief footer — it should be border-top: 2px solid var(--amber).
High
account-brief-v2 has broken color aliases: --gold = violet, --conga-purple = teal
In account-brief-v2/index.html, the token --gold maps to #8B5CF6 (violet) and --conga-purple maps to #0D9488 (teal). These are legacy template artifacts. Replace with --c-valuemap and --c-account respectively.
High
Theme support is inconsistent — only dashboard and win-room support light theme
account-brief-v2, qualify-iq, demo-brief, and value-map are dark-only. All pages should include a [data-theme="light"] override block and a theme toggle button. Use the canonical light palette tokens from Section 02.
Med
qualify-iq uses a different step indicator style (numbered circles) than the system
qualify-iq renders step navigation as large numbered circles in the page body. The canonical pattern is small dots in header-right (see Section 06). Update qualify-iq to use step dots in the header, removing the custom step nav.
Med
Footer amber border is 1px on some pages instead of 2px
qualify-iq footer uses border-top: 1px solid var(--amber-muted). Canonical spec is border-top: 2px solid var(--amber). All footer instances must use the 2px solid amber border — it is part of the protected chrome identity.
Med
Header heights are 56–64px across pages instead of a uniform 64px
sales-hub, qualify-iq, and value-map headers are 56–58px. Standardize all pages to exactly height: 64px. Adjust any padding-top on main content accordingly.
Low
Transition timing uses 0.15s/0.18s on some pages instead of --transition
Some pages use transition: all 0.15s ease or 0.18s inline. Standardize to var(--transition) everywhere to ensure consistent feel across hover states.
Low
Each page re-declares the full :root variable block (~40 lines of duplication)
Long-term: extract tokens to shared/tokens.css and shared/chrome.css, import via link tag. Short-term: at minimum ensure all pages use identical variable names and values when copying the block.
